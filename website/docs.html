Documentation

Everything you need to install termcoder, run it, and make it your own.

Overview

termcoder is an open source AI coding agent that runs in your terminal. You describe a task in plain language; it reads the relevant files, proposes and applies edits, runs commands, and reports back — asking permission before anything that changes your machine.

It ships one binary, term, and works with any model: Anthropic, OpenAI, or Google Gemini with your own key, or a local model through Ollama. A second persona, termexplorer, turns the same tool into a study assistant.

Install

You need Node.js 18 or newer. Install the CLI globally from npm:

# Windows (PowerShell or CMD), macOS, or Linux
npm install -g @termcoder/tui

That adds two equivalent commands to your path — term and termcoder. Verify the install:

term --version

Desktop app. Prefer a window? Download the Windows installer from GitHub Releases. It runs the same engine as the CLI.

First run

Open a terminal in a project folder and run:

term

Two things happen the first time:

  1. Trust prompt. termcoder asks whether you trust the current folder before it reads or runs anything in it. It remembers your answer per folder.
  2. Model setup. If no provider is configured, run /setup — it walks you through connecting a key, or points you at a local Ollama model so you can start without one.

Once a model is ready, type your request at the prompt and press Enter.

Quickstart

A first session, end to end:

# in your project folder
term

# connect a model (once)
❯ /setup

# ask for a change — it reads, edits, and can run tests
❯ add input validation to the signup form and run the tests

# review the plan before it touches files
❯ /model            # switch models any time
❯ shift+tab          # toggle Plan / Build mode

termcoder shows each tool call as it happens, collapses long output, and prints a diff for every edit before applying it.

How it works

Each turn, the agent decides which tools to call to satisfy your request. The built-in tools let it read files, list and search the tree, write and edit files, run shell commands, and fetch the web.

Anything that changes your machine — writing a file, running a command — goes through a permission check first. You can allow it once, allow it for the session, or deny it. Turn on /auto to auto-approve for the current session when you trust the task.

Long tool output is collapsed in the transcript and trimmed from the model's context as the session grows, so a long conversation does not keep re-sending everything it has already read.

Build and Plan modes

termcoder has two modes. Toggle them with shift+tab — the current mode shows at the bottom of the screen.

  • Build — the default. The agent can read, edit, write, and run commands (with your permission).
  • Plan — read-only. The agent inspects the code and proposes an approach without changing any files. Use it to review a plan before committing to it, then switch to Build.

Files and sub-agents

Two characters give the agent extra context while you type:

TypeResult
@Attach a file. A picker opens as you type a path; the file's contents go to the agent.
$Hand the task to a specialist sub-agent (reviewer, tester, debugger, architect).

Sub-agents run a focused sub-task with their own tools and report a summary back to the main session, which keeps the main context clean.

Models and providers

Open the picker with /model. It searches a live catalog of models and groups them:

  • Favorites — models you have starred (ctrl+f in the picker).
  • termcoder AI — our own termcoder/auto and termexplorer/auto.
  • Cloud — Anthropic, OpenAI, and Google models (need a key).
  • Local — models served by Ollama on your machine.

A filled dot means the model is ready to use; an open dot means it needs a key. You can also type a full provider/model id to add one that is not in the list.

termcoder/auto

The default. It routes each task by difficulty — a quick model for simple edits, a stronger one for harder work — and reviews its own changes before finishing.

Connecting a key

The easiest path is /setup. To set a key directly:

❯ /key anthropic sk-ant-...
❯ /key openai sk-...
❯ /key google AIza...

Keys are stored in your user config directory, not in the project. termcoder also reads standard environment variables if they are set:

ProviderEnvironment variable
AnthropicANTHROPIC_API_KEY
OpenAIOPENAI_API_KEY
Google GeminiGEMINI_API_KEY or GOOGLE_GENERATIVE_AI_API_KEY

Running locally

To run without any provider key, install Ollama and pull a model:

# pull a capable local coding model
ollama pull qwen2.5-coder

# then pick it in termcoder
❯ /model   # choose it under "Local"

termcoder detects running Ollama models automatically and lists them in the picker.

Custom agents

An agent is a named role with its own prompt, model, and tool permissions. Define one by adding a Markdown file with front matter to .termcoder/agents/ in your project (or the same folder under your user config for a global agent):

# .termcoder/agents/reviewer.md
---
description: Reviews changes, never edits
mode: subagent
model: anthropic/claude-sonnet-5
tools: [read, ls, grep, glob]
---
You are a careful code reviewer. Point out correctness bugs and
risky changes. Do not modify files; describe what should change.

Switch to an agent with /agent, or hand it a sub-task with $. Restricting tools is enough to make an agent read-only.

Skills

A skill is a reusable playbook the agent loads only when a task calls for it — a set of instructions, and optionally scripts or examples. Add skills under .termcoder/skills/:

# .termcoder/skills/release/SKILL.md
---
name: release
description: Cut a release — bump version, tag, changelog
---
Steps to cut a release in this repo:
1. Bump the version in package.json
2. Update CHANGELOG.md
3. Commit, tag, and push

List available skills with /skills. The agent reads a skill's full instructions the moment it decides to use it, so idle skills cost nothing.

Study mode

termcoder ships a sister persona for schoolwork. Open /model and pick termexplorer/auto. The tool switches from a coding agent to a patient tutor:

  • Explains concepts step by step, in your language.
  • Summarizes notes, articles, or a chapter into key points.
  • Builds flashcards, practice quizzes, and study plans.
  • Works through homework by showing the reasoning, not just the answer.

No programming knowledge is needed — just ask.

Configuration

Project settings live in .termcoder/ in your repo. Personal settings and secrets live in your user config directory:

LocationHolds
.termcoder/agents/Project agents.
.termcoder/commands/Project commands.
.termcoder/skills/Project skills.
AGENTS.mdAlways-on project instructions (create with /init).
user config dirAPI keys, theme, favorites, trusted folders, drafts.

Secrets stay out of git. Keys are written to your user config directory, never to the project. Keep .termcoder/ in your .gitignore if it holds anything local.

Change the theme any time with /theme; your choice is saved for next time.

SDK and server

termcoder is a monorepo. The engine is headless and can be driven from your own code, and an HTTP + WebSocket server powers the desktop app.

  • @termcoder/core — the agent engine: sessions, tools, permissions, providers.
  • @termcoder/server — the HTTP + WebSocket API.
  • @termcoder/tui — the terminal interface (the term binary).

See the docs folder on GitHub for the SDK and server API reference.

Troubleshooting

The term command isn't found

Make sure the install finished and that npm's global bin directory is on your PATH. Reopen the terminal after installing. Check with npm root -g.

It says a model needs a key

Run /setup, or set the provider's key with /key. To run without a key, use a local Ollama model (see Running locally).

An auth or quota error mid-task

The error line points you to /setup. Re-connect the provider or switch models with /model, then /retry.

Nothing happens when I run term

Confirm Node.js 18+ with node --version, then reinstall with npm install -g @termcoder/tui.

Still stuck? Open an issue on GitHub.
